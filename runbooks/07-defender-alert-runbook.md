# Microsoft Defender Alert Investigation Runbook

## Runbook Overview

This runbook provides a structured process for investigating alerts generated by Microsoft Defender Antivirus on Windows systems.

It applies to events involving:

* Malware or potentially unwanted application detections
* Suspicious scripts or executables
* Behavioral detections
* Quarantined files
* Failed or incomplete remediation
* Repeated detections
* Defender configuration changes
* Real-time protection interruption
* Wazuh or Splunk alerts based on Defender telemetry
* Controlled CyberLab validation activity

The runbook helps the analyst determine:

* What Defender detected
* Which endpoint was affected
* Which file, process, or resource triggered the alert
* Which user and process were involved
* Whether the detection was blocked or remediated
* Whether the artifact executed
* Whether persistence or follow-on activity occurred
* Whether the alert represents a true positive, false positive, approved test, or unresolved event
* Whether containment or escalation is required

## Scope

This runbook applies to:

* Microsoft Defender Antivirus
* Windows Security Protection History
* Microsoft Defender Operational events
* Defender PowerShell cmdlets
* Files, scripts, archives, memory, processes, and network resources referenced by Defender
* Wazuh alerts generated from Defender events
* Defender events indexed in Splunk

This runbook does not authorize:

* Executing a detected file
* Restoring a suspicious file from quarantine
* Disabling Defender
* Adding exclusions to bypass detection
* Reproducing malicious content in public documentation
* Uploading sensitive artifacts to public services
* Testing real malware outside an isolated, authorized lab
* Deleting evidence before collection

## Initial Analyst Questions

Determine immediately:

* Is the endpoint still connected?
* Is the detection active, remediated, or unresolved?
* Was the artifact blocked before execution?
* Does the file still exist?
* Was the file quarantined?
* Did the user attempt to restore it?

* Is the user privileged?
* Is the detection part of an approved validation exercise?
* Did Defender remain enabled?
* Are similar detections present on other endpoints?

# Severity Classification

## Informational

Use informational severity when:

* The detection is part of an approved harmless validation exercise
* Defender blocked the artifact immediately
* No execution or follow-on activity occurred
* Cleanup was completed
* Defender remained healthy

---

## Low

Use low severity when:

* Defender blocked a known unwanted but low-impact artifact
* The file did not execute
* The user and source are known
* Remediation succeeded
* No persistence, network, or privilege activity followed

---

## Medium

Use medium severity when:

* Execution status is unclear
* The file remains on disk
* Remediation is incomplete
* The artifact originated from email, browser, archive, or removable media
* The file is unsigned or unusual
* Process attribution is incomplete
* The same threat recurs

---

## High

Use high severity when:

* The file or script executed
* Defender reports remediation failure
* A privileged user or SYSTEM context is involved
* The process created child processes
* Network connections followed
* Persistence was created
* Security settings were modified
* The detection appears on multiple endpoints
* The user restored or attempted to restore the item

---

## Critical

Use critical severity when:

* Confirmed malware is actively executing
* Credential theft occurred
* Security controls were disabled
* Ransomware or destructive behavior is present
* Domain-wide changes occurred
* Multiple systems are compromised
* Privileged persistence was established
* Defender cannot contain the activity

# Initial Triage

## Step 1: Confirm the Alert Time

Record the exact timestamp and timezone from:

* Endpoint
* Defender event
* Wazuh
* Splunk
* Analyst workstation

Use an absolute timestamp.

```
YYYY-MM-DD HH:MM:SS Timezone
```

---

## Step 2: Confirm the Local Defender Event

Run on the affected endpoint from an authorized administrator session:

```
Get-WinEvent `
    -FilterHashtable @{
        LogName = "Microsoft-Windows-Windows Defender/Operational"
        Id = 1006, 1007, 1008, 1116, 1117, 1118, 1119, 1121
        StartTime = (Get-Date).AddHours(-2)
    } |
Select-Object `
    TimeCreated,
    Id,
    LevelDisplayName,
    Message
```

Search for a known threat name or filename:

```
Get-WinEvent `
    -FilterHashtable @{
        LogName = "Microsoft-Windows-Windows Defender/Operational"
        StartTime = (Get-Date).AddHours(-2)
    } |
Where-Object {
    $_.Message -match "<THREAT_NAME>|<FILENAME>"
}
```

If the event is not present locally, review:

* Wrong endpoint
* Wrong time range
* Stale SIEM event
* Field-extraction problem
* Alert generated from another Defender component
* Event-log retention

---

## Step 3: Review Protection History

On the affected endpoint:

1. Open **Windows Security**.
2. Select **Virus & threat protection**.
3. Open **Protection history**.
4. Locate the matching detection.
5. Review the threat name, time, resource, status, and action.
6. Do not select **Allow on device** or **Restore**.
7. Record sanitized evidence.

Protection History may summarize activity differently from the raw Operational log.

---

## Step 4: Review Defender Threat Records

```
Get-MpThreatDetection |
    Sort-Object InitialDetectionTime -Descending |
    Select-Object `
        InitialDetectionTime,
        LastThreatStatusChangeTime,
        ThreatID,
        ActionSuccess,
        CurrentThreatExecutionStatusID,
        Resources
```

Review known threat definitions:

```
Get-MpThreat |
    Select-Object `
        ThreatID,
        ThreatName,
        SeverityID,
        CategoryID,
        DidThreatExecute,
        IsActive
```

Available properties may differ by Defender version.

# Detection Analysis

## Review the Threat Name

The threat name may identify:

* Malware family
* Generic heuristic
* Script behavior
* Potentially unwanted application
* Suspicious archive
* Exploit behavior
* Test artifact
* Behavioral rule

A generic name may require more local telemetry than a specific family name.

---

## Review the Affected Resource

Defender may reference:

* File path
* Archive member
* Process memory
* Registry value
* Web resource
* Network share
* Email attachment
* Browser download
* PowerShell content
* Script host content

Record every resource associated with the detection.

---

## Review Defender Action

Possible outcomes include:

```
Blocked
Quarantined
Removed
Cleaned
Allowed
No action
Action failed
Pending
```

Confirm action success from both:

* Protection History
* Defender event or PowerShell records

Do not interpret a detection event alone as successful remediation.

---

## Determine Whether the Threat Executed

Use all available evidence:

* `DidThreatExecute`
* Current threat execution status
* Sysmon process creation
* Windows Event ID `4688`
* PowerShell logs
* Child processes
* Network connections
* File and registry changes
* Persistence
* User report

The absence of a process event does not always prove that no code executed, especially when telemetry is incomplete.

# File and Artifact Investigation

## Confirm Whether the File Exists

```
Test-Path -LiteralPath "<FILE_PATH>"
```

When present:

```
Get-Item -LiteralPath "<FILE_PATH>" -Force |
    Select-Object `
        FullName,
        Length,
        CreationTime,
        LastWriteTime,
        LastAccessTime,
        Attributes
```

Do not open or execute the file.

---

## Hash the Artifact

When the file remains accessible:

```
Get-FileHash `
    -LiteralPath "<FILE_PATH>" `
    -Algorithm SHA256
```

Record:

```
SHA-256:
File size:
Creation time:
Modification time:
```

Quarantined artifacts may no longer be available at the original path.

---

## Review the Digital Signature

```
Get-AuthenticodeSignature `
    -FilePath "<FILE_PATH>"
```

Record:

```
Signature status:
Signer:
Certificate subject:
Certificate issuer:
```

An unsigned file is not automatically malicious. A valid signature does not prove the file was authorized.

---

## Review File Origin

Check alternate data streams:

```
Get-Item `
    -LiteralPath "<FILE_PATH>" `
    -Stream * `
    -ErrorAction SilentlyContinue
```

A `Zone.Identifier` stream may indicate the file originated from a browser, email client, or another security zone.

---

## Preserve the Artifact Safely

Only when evidence collection is authorized and Defender has not quarantined the file:

```
Copy-Item `
    -LiteralPath "<FILE_PATH>" `
    -Destination "<EVIDENCE_PATH>\preserved-artifact" `
    -Force
```

Store suspicious artifacts in restricted evidence storage.

Do not place them in:

* Git repositories
* Shared transfer folders
* Public cloud folders
* Documentation directories

## Search Repeated Detections

```
index=windows earliest=-24h
(
    EventCode=1006 OR
    EventCode=1116
)
| eval threat=coalesce(ThreatName, "Unknown")
| eval resource=coalesce(Path, Resources, "Unknown")
| stats
    count as detections
    min(_time) as first_seen
    max(_time) as last_seen
    values(EventCode) as event_codes
    values(resource) as resources
    by host threat
| where detections > 1
| convert ctime(first_seen) ctime(last_seen)
| sort - detections
```

Repeated detections may indicate:

* Failed remediation
* Artifact recreation
* User restoration
* Persistence
* Recurring download
* Duplicate ingestion

# Common Investigation Scenarios

## Scenario 1: Approved Harmless Validation Artifact

Indicators:

* Approved exercise window
* Official harmless antivirus test artifact
* Dedicated test directory
* Defender blocks or quarantines it immediately
* No execution or follow-on activity
* Artifact removed during cleanup

Response:

* Confirm Defender event and remediation
* Validate Wazuh and Splunk ingestion
* Confirm artifact is absent
* Classify as authorized test activity

Do not reproduce the artifact content in public documentation.

---

## Scenario 2: Browser Download Blocked

Indicators:

* Browser parent or download path
* File detected before execution
* Defender remediation succeeds
* No child process or network activity follows

Response:

* Review download source and user action
* Confirm no other artifacts were downloaded
* Confirm the file did not execute
* Provide user guidance where appropriate
* Close after validation

---

## Scenario 3: Email Attachment Detection

Indicators:

* Email client or attachment cache path
* Archive or document-related resource
* Defender blocks extraction or execution
* User reports opening a message or attachment

Response:

* Preserve email metadata where available
* Do not reopen the attachment
* Review Office or script child processes
* Review other recipients or endpoints
* Escalate if execution occurred

---

## Scenario 4: Defender Remediation Failure

Indicators:

* Event ID `1008`, `1118`, or `1119`
* Artifact remains present
* Repeated detection
* File locked by a process
* Permission or service problem

Response:

* Increase severity
* Isolate the endpoint when warranted
* Preserve the file and process evidence
* Stop harmful processes after evidence collection
* Run an approved scan
* Confirm removal and health

---

## Scenario 5: Threat Executed Before Detection

Indicators:

* Process creation before Defender detection
* Child processes
* File, registry, or service changes
* DNS or network activity
* `DidThreatExecute` or other evidence indicates execution

Response:

* Treat as high severity
* Isolate the endpoint
* Review persistence and credentials
* Preserve volatile and disk evidence
* Escalate for incident response

---

## Scenario 6: User Restored a Quarantined Item

Indicators:

* Quarantine restoration event
* Artifact returns to original or new path
* User or administrator action is identified
* Detection repeats

Response:

* Escalate based on artifact risk
* Recontain the file
* Review why restoration occurred
* Review user and administrator activity
* Confirm no execution followed

---

## Scenario 7: Defender Exclusion Added

Indicators:

* Event ID `5007`
* New path, process, or extension exclusion
* Change follows suspicious PowerShell
* Detection stops unexpectedly

Response:

* Treat as high severity
* Preserve current and previous configuration
* Identify the actor and process
* Remove unauthorized exclusion
* Review the excluded location for artifacts
* Validate Defender health

---

## Scenario 8: Potential False Positive

Indicators:

* Known approved software
* Valid signature
* Trusted installation source
* Reproducible application behavior
* No suspicious process or network activity
* Vendor or internal review supports legitimacy

Response:

* Preserve the detection
* Validate the file hash and source
* Do not add a broad exclusion
* Use the narrowest approved exception only after formal review
* Continue monitoring

# Containment

## When Containment Is Not Required

Containment may not be necessary when:

* The activity is an approved validation exercise
* Defender blocked the artifact before execution
* Remediation succeeded
* No suspicious follow-on activity exists
* Defender remains healthy

Document why containment was unnecessary.

---

## Endpoint Containment

Depending on severity:

* Disconnect the VM’s network adapter
* Disable the affected network adapter
* Isolate through an endpoint platform
* End unauthorized remote sessions
* Block external connectivity
* Preserve volatile evidence

Do not power off the endpoint before considering volatile evidence.

---

## Process Containment

When harmful activity is active and evidence has been preserved:

```
Stop-Process `
    -Id <PROCESS_ID> `
    -Force
```

Record:

```

Administrator:
Reason:

```

---

## Account Containment

With appropriate authorization:

* Disable or restrict the affected user
* Reset credentials
* Revoke sessions
* Remove unauthorized privilege
* Restrict remote access
* Contain the actor account that modified Defender

---

## Network Containment

* Block known malicious destination
* Block related domain
* Restrict the affected port
* Disable unnecessary remote service
* Limit endpoint access to required internal systems

Use narrow controls with documented rollback steps.

# Eradication

## Remove or Quarantine the Artifact

Prefer Defender-managed remediation.

Do not manually restore or execute the item.

Confirm:

* Original path is clear
* Quarantine status is known
* Related copies are removed
* Archive members are reviewed
* Startup and temporary locations are checked

---

## Run an Approved Scan

Update signatures:

```
Update-MpSignature
```

Run a quick scan:

```
Start-MpScan -ScanType QuickScan
```

Use a full scan when risk and time justify it:

```
Start-MpScan -ScanType FullScan
```

Do not rely on a scan alone when execution or persistence is confirmed.

---

## Remove Persistence

Depending on findings:

* Delete unauthorized scheduled tasks
* Remove unauthorized services
* Remove registry autoruns
* Remove startup-folder artifacts
* Restore modified PowerShell profiles
* Remove malicious scripts
* Correct Group Policy or deployment settings

Preserve evidence before removal.

---

## Restore Defender Configuration

Confirm:

* Real-time protection enabled
* Behavior monitoring enabled
* Script scanning enabled
* No unauthorized exclusions
* Signature updates working
* Defender services healthy

Do not create broad exclusions to suppress recurring alerts.

---

## Address Compromised Accounts

* Disable or restrict accounts
* Reset credentials
* Revoke active sessions
* Review successful logons
* Review group changes
* Review additional endpoints
* Restore access only after scope is understood

# MITRE ATT&CK Context

A Defender alert should not be mapped to ATT&CK solely because a detection exists.

Apply ATT&CK techniques based on observed behavior, such as:

* T1059.001 — PowerShell
* T1105 — Ingress Tool Transfer
* T1204 — User Execution
* T1053.005 — Scheduled Task
* T1543.003 — Windows Service
* T1547 — Boot or Logon Autostart Execution
* T1562.001 — Impair Defenses
* T1003 — OS Credential Dumping
* T1071 — Application Layer Protocol
* T1070 — Indicator Removal

The official harmless antivirus test artifact does not establish adversary behavior.

# Summary

This runbook provides a repeatable process for investigating Microsoft Defender Antivirus alerts.

A complete investigation should determine:

* What Defender detected
* Which endpoint and user were involved
* Which resource triggered the detection
* Whether the artifact existed or was quarantined
* Whether it executed
* Which parent and child processes were involved
* Whether network activity occurred
* Whether persistence was created
* Whether Defender remediation succeeded
* Whether Defender configuration was modified
* Whether the activity was an approved test, false positive, blocked threat, or confirmed incident
* Whether containment and eradication were required
* Whether Defender and monitoring returned to a healthy state

The investigation is complete only after the detection is classified, required response actions are completed, evidence is preserved, Defender health is validated, monitoring confirms the intended endpoint state, and detection improvements are documented.
